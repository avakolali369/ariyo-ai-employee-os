# Ariyo Company Operating System Index

## Core Contract

Ariyo Company Operating System Index is not a folder index, roadmap summary, founder memo, product checklist, or investor appendix.

It is the final governing index that connects Ariyo's product system, AI Employee operating model, Marketplace ecosystem, Enterprise governance layer, frontend implementation system, Codex execution protocol, MVP scope, demo narrative, pilot learning loop, production release gates, investor proof system, fundraising motion, executive KPI system, and post-funding governance rhythm into one company operating system.

Ariyo must not run as disconnected documents.

Ariyo must run as a governed operating system where every skill has a role, every role has an execution order, every execution order has a decision gate, every decision gate has evidence, every evidence object has an owner, and every owner has a next action.

---

## Non-Negotiable Operating Law

Codex, AI-assisted builders, founders, designers, frontend engineers, product owners, operators, sales teams, customer-success teams, investor-relations owners, and governance reviewers must use this file as the **highest-level navigation map** for the Ariyo skill system.

This file does not replace specialist skills.

This file tells the team which specialist skill to use, when to use it, what it depends on, what it must produce, and which downstream decisions it enables.

---

## Global RTL/LTR Layout Law

Persian/Farsi and Arabic interfaces must be RTL and right-aligned across all UI elements.

English and European LTR interfaces must be LTR and left-aligned across all UI elements.

This law applies to every public page, product surface, dashboard, marketplace screen, enterprise console, internal admin surface, customer portal, investor surface, demo page, report, email-like interface, exported PDF-like view, and frontend implementation.

If any visual, frontend, product, or mockup instruction conflicts with this law, this law wins.

## Company Operating System Layers

Ariyo has nine operating layers.

### Layer 1: Company Narrative & Strategy

This layer explains what Ariyo is, why it exists, which market it serves, what value it creates, and how it should be communicated.

Primary outputs:

- category narrative
- strategic positioning
- MVP boundary
- roadmap logic
- investor story
- website and GTM narrative

### Layer 2: Product & Surface System

This layer turns strategy into visible product experiences.

Primary outputs:

- product surface map
- public website hierarchy
- app shell
- dashboard surfaces
- AI Employee surfaces
- marketplace surfaces
- enterprise admin surfaces
- trust and executive surfaces

### Layer 3: Design System & Frontend System

This layer turns product surfaces into reusable design and code.

Primary outputs:

- Figma foundations
- tokens
- typography
- spacing
- component library
- responsive behavior
- RTL/LTR variants
- frontend sprint plan
- test plan

### Layer 4: Data, Mock API, Backend Contract System

This layer keeps demo, frontend, backend, API, and production migration aligned.

Primary outputs:

- demo seed data
- typed fixtures
- mock API contracts
- backend contract review
- migration gates
- production-ready API boundaries

### Layer 5: AI Employee Runtime System

This layer defines how AI Employees are created, governed, observed, tested, cost-controlled, and improved.

Primary outputs:

- AI Employee builder
- memory/context
- permissions
- simulation
- evaluation
- incident response
- versioning
- human review
- cost control

### Layer 6: Marketplace & Ecosystem System

This layer defines how templates, creators, partners, private catalogs, installation, trust, reviews, monetization, and compliance work.

Primary outputs:

- marketplace discovery
- template detail
- install flow
- creator portal
- partner portal
- template governance
- marketplace health
- enterprise procurement

### Layer 7: Enterprise Trust & Governance System

This layer defines why enterprise buyers can trust, govern, audit, approve, and scale Ariyo.

Primary outputs:

- enterprise admin
- SSO/SCIM
- approval workflows
- policy engine
- data residency
- trust center
- compliance evidence
- public status

### Layer 8: Customer Success, Pilot, Release & Operations System

This layer turns product usage into customer outcomes, pilot proof, readiness gates, and operating discipline.

Primary outputs:

- onboarding
- customer health
- success playbooks
- pilot onboarding
- feedback-to-roadmap
- production readiness
- release gate
- internal ops

### Layer 9: Fundraising, Executive Governance & Company Rhythm System

This layer turns company progress into investor confidence, board discipline, capital accountability, and post-funding execution.

Primary outputs:

- investor proof
- data room
- fundraising outreach
- investor CRM
- weekly update
- term readiness
- closing plan
- post-funding operating plan
- board governance rhythm
- executive KPI dashboard

## Source-of-Truth Hierarchy

When two skills appear to conflict, use this hierarchy.

1. Safety, privacy, security, compliance, and legal constraints win.
2. RTL/LTR layout law wins for localization and interface direction.
3. `company-operating-system-index.md` wins for operating navigation.
4. `codex-execution-master-prompt.md` wins for Codex behavior.
5. `mvp-build-scope-lock.md` wins for MVP build boundary.
6. `production-release-gate-checklist.md` wins for release readiness.
7. Surface-specific mockup specs win for page-level layout.
8. API contracts win for data shape.
9. Demo data packs win for mock values.
10. Roadmap and strategy docs guide prioritization, not local implementation details.

---

## Codex Stop Rules

Codex must stop and report if:

- a requested surface is not inside MVP/demo scope;
- a page requires backend behavior not represented in mock API contracts;
- a requested feature conflicts with security, privacy, compliance, or enterprise governance rules;
- RTL/LTR behavior is not defined for the requested surface;
- redaction requirements are unclear for investor, board, enterprise, or internal modes;
- demo data would create misleading product claims;
- release gates have not passed;
- required skill files are missing;
- a task would create a one-off component instead of using the design system;
- an implementation would make Ariyo look like a chatbot, prompt library, generic automation tool, or generic SaaS dashboard.

## Company OS Completion Criteria

Ariyo's Company Operating System is considered usable when:

- Codex can identify which skill to read first for any work unit;
- Product surfaces are not designed outside the surface map;
- MVP implementation is locked;
- demo data is consistent across surfaces;
- mock API and backend contracts are aligned;
- RTL/LTR is enforced globally;
- release gates exist before production claims;
- pilot evidence can become roadmap decisions;
- investor proof can be traced to customer/product evidence;
- executive KPI views are redaction-safe and board-ready;
- post-funding operations convert capital into measurable progress.

Ariyo is not a collection of screens, prompts, agents, workflows, templates, dashboards, investor materials, or operational checklists.
