# Ariyo Final Codex Handoff Pack

## Core Contract

Ariyo Final Codex Handoff Pack is not a README, compressed archive note, prompt snippet, or informal developer instruction.

It is the final governed handoff package that tells Codex, frontend engineers, AI-assisted builders, product owners, and technical operators exactly how to begin, sequence, execute, validate, report, and stop when implementing Ariyo's full skill system.

Codex must not guess the order of work.
Codex must not execute skills alphabetically.
Codex must not implement disconnected screens.
Codex must not treat mockups, MVP, pilot, production, investor, and governance documents as separate products.

Codex must execute Ariyo as one company operating system.

---

## Purpose

This file is the final operational bridge between the Ariyo `.skills` package and real implementation work.

It answers six questions:

1. Which files must Codex read first?
2. Which files define source-of-truth hierarchy?
3. Which product surfaces must be built first?
4. Which files constrain design, data, API, security, and release readiness?
5. What must Codex report after every implementation unit?
6. When must Codex stop instead of continuing with unsafe or unclear execution?

---

## Non-Negotiable Execution Laws

### 1. Company OS Law

Ariyo must be implemented as an AI Employee Operating System, not as disconnected SaaS pages.

Every screen, component, API, demo dataset, mock flow, release gate, and investor artifact must map to the Company Operating System Index.

### 2. Skill Order Law

Codex must follow the boot order in this file and the complete execution order in `codex-execution-master-prompt.md`, `mockup-implementation-roadmap.md`, and `company-operating-system-index.md`.

Alphabetical implementation is forbidden.

### 3. Scope Lock Law

During MVP execution, Codex must obey `mvp-build-scope-lock.md`.

If a feature is marked Demo Real, Deferred, Constraint Only, or Forbidden During MVP, Codex must not silently build it as production functionality.

### 4. RTL/LTR Layout Law

Persian/Farsi and Arabic interfaces must be RTL and right-aligned across all UI elements.

English and European LTR interfaces must be LTR and left-aligned across all UI elements.

This applies to every surface, component, table, card, chart label, sidebar, modal, form, navigation element, dashboard, export, notification, and error state.

### 5. Evidence Law

Codex must not create fake proof, fake traction, fake compliance, fake production readiness, fake investor evidence, or fake pilot outcomes.

Demo data is allowed only when it is labeled and structured according to `demo-data-seed-pack.md`, `frontend-demo-data-implementation-guide.md`, and related mock API rules.

### 6. Safety & Governance Law

Any surface involving permissions, AI actions, approvals, customer data, marketplace install, enterprise governance, investor-safe reporting, board export, or production launch must respect security, privacy, redaction, audit, and release gate skills.

## Codex Implementation Modes

Codex may operate in these modes only:

### Mode 1: Foundation Mode

Used for project structure, design tokens, app shell, routing, mock data, API boundary, theme, localization, RTL/LTR, and shared components.

### Mode 2: Surface Mode

Used to implement a defined product surface from its mockup spec.

### Mode 3: Component Mode

Used to build reusable design-system components.

### Mode 4: Data Mode

Used to implement demo fixtures, selectors, mock API endpoints, and state scenarios.

### Mode 5: Test Mode

Used to add unit, integration, accessibility, RTL/LTR, permission, and export tests.

### Mode 6: Release Gate Mode

Used to verify readiness against release gates and stop rules.

### Mode 7: Documentation Mode

Used to produce work unit reports, implementation notes, or handoff documentation.

## Codex Stop Rules

Codex must stop and report instead of continuing when:

1. A required source-of-truth skill is missing.
2. Two skill files conflict on a safety, permission, data, release, or scope issue.
3. A requested feature is outside `mvp-build-scope-lock.md`.
4. A UI surface requires real backend behavior that only exists as demo data.
5. A board/investor/export surface would expose unsafe or unredacted information.
6. A Persian/Arabic UI cannot be confidently implemented as RTL.
7. A data object has no type or no stable mock API contract.
8. A release gate fails.
9. A test plan requires coverage not yet implemented.
10. Codex would need to invent legal, compliance, investor, financial, or customer proof.

## Source-of-Truth Hierarchy

When files conflict, Codex must resolve them in this order:

1. `company-operating-system-index.md`

6. Product surface and mockup specs
7. Frontend implementation and test plans
8. Demo data and mock API guides
9. Growth, fundraising, and investor proof systems
10. Older tactical skills

Safety, privacy, permissions, RTL/LTR, and release gates always override visual preference.

## Definition of Done

A Codex implementation unit is done only when:

- it names the skill files used
- it follows the source-of-truth hierarchy
- it respects MVP scope
- it uses typed data
- it respects mock API contracts
- it implements required states
- it passes RTL/LTR expectations
- it passes accessibility basics
- it does not invent evidence
- it reports blockers honestly
- it provides next work unit recommendation

---

## Forbidden Codex Behaviors

Codex must not:

- build pages alphabetically
- treat skill files as optional inspiration
- ignore release gates
- hardcode random fake metrics
- create untyped demo data
- skip error/loading/empty states
- ignore RTL/LTR
- create investor proof without evidence
- expose internal data in investor-safe views
- build production-only features during MVP without scope approval
- silently resolve governance conflicts
- produce screens that look premium but do not match Ariyo's operating model

## Final Handoff Instruction

When Codex receives the Ariyo `.skills` package, it must first read this file and then answer:

1. What is the Ariyo product being built?
2. What is the MVP scope?
3. What is the first implementation sprint?
4. What files will be read before coding?
5. What stop rules are active?
6. What is the next work unit report format?

Only after answering those questions should Codex begin implementation.

Codex is not building a demo website.
